## 1. Introduction and Scope

### 1.1 Definition of Computer-Use Agents

A Computer-Use Agent (CUA) is software that autonomously or semi-autonomously interacts with graphical user interfaces to accomplish tasks. CUAs combine:

- **Perception**: Screen capture, OCR, element detection
- **Reasoning**: Task planning, error recovery, context understanding
- **Action**: Input injection (keyboard, mouse), API calls, process management

### 1.2 Scope of Research

| Category | Included | Excluded |
|----------|----------|----------|
| **Platforms** | macOS, Linux, Windows | Mobile (iOS, Android) |
| **Interfaces** | GUI, CLI | Voice-only, hardware |
| **Techniques** | Native APIs, AI/ML | Pure RPA (no perception) |
| **Timeframe** | 2020-2026 | Legacy (< 2020) |

### 1.3 Research Methodology

- Literature review (ACM, IEEE, arXiv)
- Source code analysis (GitHub, GitLab)
- Commercial documentation analysis
- Performance benchmarking (where possible)
- Expert interviews (industry practitioners)

## 2. Academic Foundations

### 2.1 Web Automation

#### 2.1.1 WebArena (Zhou et al., 2024)

WebArena [1] established the first realistic web environment for autonomous agents. Key contributions:

- **Realistic environment**: Full websites (GitLab, Shopify, etc.) in containers
- **Benchmark suite**: 812 tasks across 5 domains
- **Metrics**: Task success rate, step efficiency, safety

```
WebArena Architecture:
┌─────────────────────────────────────────────────────────────┐
│                      Agent (LLM + Tools)                    │
├─────────────────────────────────────────────────────────────┤
│         Thought ──▶ Action ──▶ Observation                 │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │  visit   │  │  search  │  │  click   │  │  type    │    │
│  │  scroll  │  │ find_loc │  │ go_back  │  │ stop     │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
├─────────────────────────────────────────────────────────────┤
│                Browser (Playwright/Selenium)                │
├─────────────────────────────────────────────────────────────┤
│              Real Websites (containerized)                    │
└─────────────────────────────────────────────────────────────┘
```

**Results**: GPT-4 achieved 14.9% task success rate (human: 78.2%). Gap indicates significant opportunity for improvement.

#### 2.1.2 Mind2Web (Deng et al., 2023)

Mind2Web [2] focused on generalist web agents:

- **Dataset**: 137K tasks from 137 websites
- **Cross-task generalization**: Train on some sites, test on others
- **Atomic operations**: Click, type, scroll, select

### 2.2 GUI Automation Research

#### 2.2.1 OSWorld (Xie et al., 2024)

OSWorld [3] extended web automation to full OS environments:

```
OSWorld vs WebArena:
┌────────────────┬─────────────────────────┬─────────────────────────┐
│    Aspect      │        WebArena         │         OSWorld         │
├────────────────┼─────────────────────────┼─────────────────────────┤
│ Scope          │ Browser only            │ Full OS (Ubuntu)        │
│ Environment    │ Containerized websites  │ VM with full desktop    │
│ Input methods  │ Browser APIs            │ Native OS APIs          │
│ Observation    │ HTML/DOM                │ Screenshots (pixel)     │
│ Tasks          │ 812 web tasks           │ 369 OS tasks            │
│ Success rate   │ 14.9% (GPT-4)           │ 12.1% (GPT-4)           │
└────────────────┴─────────────────────────┴─────────────────────────┘
```

**Key insight**: Operating system automation is harder than web automation due to:
- Lower-level abstractions (pixels vs DOM)
- State space explosion (any application vs structured websites)
- Permission and security boundaries

#### 2.2.2 GUI-World (not published)

Several unpublished works explore GUI automation:

- **AgentStudio**: macOS-focused CUA environment
- **WindowsAgentArena**: Windows-specific benchmark
- **LinuxGym**: Linux desktop automation

### 2.3 Foundation Models for UI Understanding

#### 2.3.1 CogAgent (Hong et al., 2023)

CogAgent [4] introduced high-resolution visual understanding for GUI:

- **Architecture**: Visual encoder + LLM
- **Resolution**: 1120x1120 (vs 224x224 in standard ViT)
- **Performance**: SOTA on multiple GUI benchmarks

```

┌─────────────────────────────────────────────────────────────────┐

│  │  Cross-     │  224x224 (standard) + 1120x1120 (high-res)    │
│  │  Resolution │  Cross-attention between scales                 │
│  │  Adapter    │                                                │
│  └──────┬──────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌─────────────┐    ┌─────────────┐                             │
│  │   ViT       │───▶│    LLM      │ (Vicuna-7B/13B)              │
│  │ (EVA2-CLIP) │    │  (decoder)  │                             │
│  └─────────────┘    └──────┬──────┘                             │
│                             │                                    │
│                             ▼                                    │
│                       Action prediction                          │
│                       (click, type, scroll)                      │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.3.2 SeeClick (Cheng et al., 2024)

SeeClick [5] focused on GUI grounding (mapping natural language to screen coordinates):

- **Pre-training**: Web screenshots + HTML structure
- **Grounding accuracy**: 85.6% on ScreenSpot benchmark
- **Generalization**: Cross-platform (web, mobile, desktop)

### 2.4 Tool Learning and API Calling

#### 2.4.1 ToolLLM (Qin et al., 2023)

ToolLLM [6] demonstrated LLM ability to learn and use APIs:

- **Dataset**: 16,464 APIs from RapidAPI
- **Training**: Instruction-following with tool use
- **Result**: GPT-4-level tool use with smaller models

#### 2.4.2 Gorilla (Patil et al., 2023)

Gorilla [7] specialized in API call generation:

- **1,600+ APIs** from ML models, HuggingFace, TorchHub
- **Retrieval-aware training**: API docs as context
- **Output**: Executable API calls with parameters

## 4. Open-Source Frameworks

### 4.1 Scrapy / Playwright / Selenium

Traditional web automation tools form the foundation of many CUAs:

| Framework | Type | Language | Maintenance | Best For |
|-----------|------|----------|-------------|----------|
| Selenium | Browser | Multi | Active | Legacy compatibility |
| Playwright | Browser | Multi | Very Active | Modern web apps |
| Puppeteer | Browser | Node | Active | Chrome automation |
| Scrapy | Web scraping | Python | Active | Data extraction |

### 4.2 PyAutoGUI / Autopy

Native GUI automation libraries:

```python
# PyAutoGUI example
import pyautogui

# Screenshot
screenshot = pyautogui.screenshot()

# Mouse
pyautogui.moveTo(100, 200, duration=0.5)
pyautogui.click()

# Keyboard
pyautogui.typewrite('Hello, World!', interval=0.1)
pyautogui.keyDown('ctrl')
pyautogui.keyDown('c')
pyautogui.keyUp('c')
pyautogui.keyUp('ctrl')
```

**Limitations**:
- No async/await support
- Single-threaded
- Platform-specific quirks
- No type safety
- Hard to test

### 4.3 robotgo

Go-based cross-platform automation:

```go
// robotgo example
package main

import (
    "github.com/go-vgo/robotgo"
)

func main() {
    // Screenshot
    bitmap := robotgo.CaptureScreen()
    
    // Mouse
    robotgo.Move(100, 200)
    robotgo.Click()
    
    // Keyboard
    robotgo.TypeStr("Hello, World!")
    robotgo.KeyTap("ctrl", "c")
}
```

**Strengths**:
- Native performance
- Cross-platform
- Active development

**Weaknesses**:
- CGO dependencies
- CGO cross-compilation issues
- Memory management complexity

### 4.4 clay

## 5. Capture Technologies

### 5.1 Windows

#### 5.1.1 Windows Graphics Capture (WGC)

**API**: `Windows.Graphics.Capture`

```
WGC Architecture:
┌─────────────────────────────────────────────────────────────┐
│                    Application                              │
│  ┌─────────────────────────────────────────────────────┐  │
│  │          GraphicsCapturePicker                      │  │
│  │  (User selects window/display to capture)            │  │
│  └─────────────────────┬───────────────────────────────┘  │
│                        │                                    │
│                        ▼                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           GraphicsCaptureItem                       │  │
│  │  (Represents capture target: window or display)      │  │
│  └─────────────────────┬───────────────────────────────┘  │
│                        │                                    │
│                        ▼                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           Direct3D11CaptureFramePool                │  │
│  │  (Manages frame buffers)                            │  │
│  └─────────────────────┬───────────────────────────────┘  │
│                        │                                    │
│                        ▼                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           IDirect3D11SurfaceImageSource             │  │
│  │  (Hardware-accelerated texture)                      │  │
│  └─────────────────────┬───────────────────────────────┘  │
│                        │                                    │
│                        ▼                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           SoftwareBitmap (optional)                 │  │
│  │  (CPU-readable copy for encoding)                    │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**Requirements**:
- Windows 10 version 1903+ (May 2019 Update)
- Windows.Graphics.Capture contract

**Performance**:
- Hardware-accelerated (GPU)
- ~1-2ms capture latency
- ~60 FPS possible

**Limitations**:
- Cannot capture protected content (DRM)
- Requires window handle for window capture
- No cursor capture in some modes

#### 5.1.2 Desktop Duplication API (DXGI)

**API**: `IDXGIOutputDuplication`

**Use case**: Full-screen capture, gaming

**Performance**: Very high (direct GPU access)

**Limitations**:
- Desktop only (no window-specific)
- No hardware cursor
- UAC prompts block capture

#### 5.1.3 BitBlt / PrintWindow (Legacy)

**API**: GDI `BitBlt`, `PrintWindow`

**Status**: Legacy, slow, unreliable

**Use**: Fallback only

### 5.2 Linux

#### 5.2.1 X11

**API**: `XGetImage`, `XShmGetImage` (shared memory)

```c
// X11 capture example
Display* display = XOpenDisplay(NULL);
Window root = DefaultRootWindow(display);

XImage* image = XGetImage(
    display, root,
    0, 0, width, height,
    AllPlanes, ZPixmap
);

// Convert to desired format
// ...

XDestroyImage(image);
XCloseDisplay(display);
```

**Performance**:
- XShm: ~5-10ms for 1920x1080
- XGetImage: ~20-50ms

**Status**: Stable, widely supported

#### 5.2.2 PipeWire / xdg-desktop-portal

**API**: `org.freedesktop.portal.ScreenCast`


### 5.3 macOS

```objc
// macOS CoreGraphics capture
CGImageRef screenshot = CGDisplayCreateImage(kCGDirectMainDisplay);
```

## 6. Input Injection Methods

### 6.1 Windows

#### 6.1.1 SendInput

**API**: `win32::UI::Input::KeyboardAndMouse::SendInput`

```rust
// Windows SendInput example
use windows::Win32::UI::Input::KeyboardAndMouse::{
    SendInput, INPUT, INPUT_MOUSE, MOUSEEVENTF_MOVE, MOUSEINPUT,
};

let input = INPUT {
    r#type: INPUT_MOUSE,
    Anonymous: INPUT_0 {
        mi: MOUSEINPUT {
            dx: 100,
            dy: 200,
            mouseData: 0,
            dwFlags: MOUSEEVENTF_MOVE,
            time: 0,
            dwExtraInfo: 0,
        },
    },
};

unsafe {
    SendInput(&[input], std::mem::size_of::<INPUT>() as i32);
}
```

**Features**:
- Low-level input
- Works with most applications
- Games, DirectX

**Limitations**:
- Requires foreground window for some apps
- UIPI (User Interface Privilege Isolation) blocks cross-elevation

#### 6.1.2 PostMessage

**API**: `PostMessage`, `SendMessage`

**Use case**: Sending messages to specific windows

**Limitations**:
- Many applications ignore messages
- Synchronization issues

### 6.2 Linux

#### 6.2.1 uinput

**API**: `/dev/uinput` (kernel interface)

```rust
// Linux uinput example (simplified)
use std::fs::OpenOptions;
use std::os::unix::io::AsRawFd;

let uinput = OpenOptions::new()
    .write(true)
    .open("/dev/uinput")?;

// Setup virtual device
// ioctl(UI_SET_EVBIT, EV_KEY)
// ioctl(UI_SET_KEYBIT, KEY_A)

// Write event
let event = input_event {
    time: timeval { tv_sec: 0, tv_usec: 0 },

// Send
// write(uinput_fd, &event, sizeof(event))
```

**Requirements**:
- udev rules or root
- CAP_SYS_ADMIN capability (without root)

```bash
# /etc/udev/rules.d/99-uinput.rules
KERNEL=="uinput", MODE="0660", GROUP="uinput"
```

#### 6.2.3 evdev

### 6.3 macOS

#### 6.3.1 CGEvent (CoreGraphics)

**API**: `CGEventCreateMouseEvent`, `CGEventPost`

```rust
// macOS CGEvent example (via core-graphics crate)
use core_graphics::event::{CGEvent, CGEventType, CGEventFlags};
use core_graphics::event_source::{CGEventSource, CGEventSourceStateID};

let source = CGEventSource::new(CGEventSourceStateID::CombinedSessionState)?;

// Move mouse
let event = CGEvent::new_mouse_event(
    source,
    CGEventType::MouseMoved,
    CGPoint::new(100.0, 200.0),
    CGMouseButton::Left,
)?;
event.post(CGEventTapLocation::HID);

// Click
let event = CGEvent::new_mouse_event(
    source,
    CGEventType::LeftMouseDown,
    CGPoint::new(100.0, 200.0),
    CGMouseButton::Left,
)?;
event.post(CGEventTapLocation::HID);
```

**Requirements**:
- Accessibility permission (TCC)
- User approval on first use

#### 6.3.2 AppleScript

**API**: `osascript`, `NSAppleScript`

## 7. IPC and Protocol Standards

### 7.1 JSON-RPC 2.0

**Specification**: [jsonrpc.org](https://www.jsonrpc.org/specification)

**bare-cua Implementation**:
```
Transport: Newline-delimited JSON over stdio
Encoding: UTF-8
Content-Type: application/json-rpc

Request:
{"jsonrpc":"2.0","id":1,"method":"screenshot","params":{"monitor":0}}

Response:
{"jsonrpc":"2.0","id":1,"result":{"data":"base64...","width":1920}}

Error:
{"jsonrpc":"2.0","id":1,"error":{"code":-32603,"message":"..."}}
```

**Error Codes**:
| Code | Message | Meaning |
|------|-----------|---------|
| -32700 | Parse error | Invalid JSON |
| -32600 | Invalid Request | JSON is not a valid request |
| -32601 | Method not found | Method doesn't exist |
| -32602 | Invalid params | Invalid method parameters |
| -32603 | Internal error | Internal error |
| -32000 to -32099 | Server error | Implementation-defined |

### 7.2 OpenRPC

**Specification**: [open-rpc.org](https://spec.open-rpc.org/)

**Purpose**: Machine-readable API specification (like OpenAPI for REST)

**bare-cua OpenRPC Document**:
```json
{
  "openrpc": "1.2.6",
  "info": { "title": "bare-cua-native", "version": "0.1.0" },
  "methods": [
    {
      "name": "screenshot",
      "params": [...],
      "result": { ... }
    }
  ]
}
```

**Benefits**:
- Auto-generated documentation
- Client generation
- Validation

### 7.3 gRPC

**Protocol**: HTTP/2 + Protocol Buffers

**Comparison with JSON-RPC**:

| Aspect | gRPC | JSON-RPC over stdio |
|--------|------|---------------------|
| Transport | HTTP/2 | stdio |
| Serialization | Binary (Protobuf) | JSON (text) |
| Streaming | Bidirectional | None (request/response) |
| Schema | Required (.proto) | Optional (OpenRPC) |
| Port binding | Required | None |
| Browser support | gRPC-Web | N/A |
| Debuggability | Hard (binary) | Easy (text) |

**Why bare-cua chose JSON-RPC**:
1. No port conflicts
2. Simple to debug
3. No code generation needed
4. Unix philosophy (pipes)
5. Language agnostic

### 7.4 WebSocket

**Protocol**: TCP with HTTP upgrade

**Use case**: Remote control, browser integration

**Consideration for bare-cua**:
- Could be added as alternative transport
- Enables web-based clients
- Requires port binding

### 7.5 Standard Streams (stdio)

**Unix Philosophy**:
- stdin: Input stream
- stdout: Output stream
- stderr: Error/diagnostic stream

**bare-cua mapping**:
- stdin: JSON-RPC requests
- stdout: JSON-RPC responses
- stderr: Structured logs (JSON)

**Benefits**:
- No network configuration
- Works through SSH
- Container-friendly
- Shell-scriptable

## 8. Cross-Platform Abstractions

### 8.1 Hexagonal Architecture (Ports and Adapters)

**Pattern**: Alistair Cockburn's Hexagonal Architecture [12]

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Hexagonal Architecture                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                          ┌──────────────┐                                   │
│                          │   Client     │                                   │
│                          │  (driving)   │                                   │
│                          └──────┬───────┘                                   │
│                                 │                                           │
│  ┌───────────────────────────────┼───────────────────────────────────────┐   │
│  │                               │         Application Core               │   │
│  │                               ▼                                      │   │
│  │  ┌────────────────────────────────────────────────────────────────┐  │   │
│  │  │                     Domain (business logic)                    │  │   │
│  │  │  - Pure types, no external dependencies                      │  │   │
│  │  │  - Frame, WindowInfo, ProcessHandle, etc.                    │  │   │
│  │  └────────────────────────────────────────────────────────────────┘  │   │
│  │                               │                                      │   │
│  │                               ▼                                      │   │
│  │  ┌────────────────────────────────────────────────────────────────┐  │   │
│  │  │                      Ports (interfaces)                        │  │   │
│  │  │  - CapturePort, InputPort, WindowPort, ProcessPort, etc.       │  │   │
│  │  └────────────────────────────────────────────────────────────────┘  │   │
│  │                               │                                      │   │
│  └───────────────────────────────┼──────────────────────────────────────┘   │
│                                  │                                           │
│  ┌───────────────────────────────┼───────────────────────────────────────┐   │
│  │                               ▼         Adapters (driven)            │   │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐               │   │
│  │  │   Windows    │  │    Linux     │  │    macOS     │               │   │
│  │  │  ┌────────┐  │  │  ┌────────┐  │  │  ┌────────┐  │               │   │
│  │  │  │   WGC  │  │  │  │  X11   │  │  │  │CoreGfx │  │               │   │
│  │  │  │SendInp │  │  │  │ uinput │  │  │  │CGEvent │  │               │   │
│  │  │  └────────┘  │  │  └────────┘  │  │  └────────┘  │               │   │
│  │  └──────────────┘  └──────────────┘  └──────────────┘               │   │
│  │                                                                    │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Benefits**:
1. **Testability**: Domain logic independent of platform
2. **Flexibility**: Swap adapters without changing core
3. **Clarity**: Dependencies point inward only
4. **Maintainability**: Platform code isolated

**Crate Structure**:

### 8.3 Trait Design

**Port Traits**:
```rust
#[async_trait]
pub trait CapturePort: Send + Sync {
    async fn capture_display(&self, monitor: u32) -> Result<Frame, CaptureError>;
    async fn capture_window(&self, title: Option<&str>) -> Result<Frame, CaptureError>;
}

#[async_trait]
pub trait InputPort: Send + Sync {
    async fn key_event(&self, key: Key, action: KeyAction) -> Result<(), InputError>;
    async fn type_text(&self, text: &str) -> Result<(), InputError>;
    async fn mouse_event(&self, event: MouseEvent) -> Result<(), InputError>;
}
```

**Design Principles**:
1. **Async-first**: All operations are async for non-blocking I/O
2. **Send + Sync**: Thread-safe for concurrent use
3. **Error propagation**: Domain-specific error types
4. **Zero-cost**: Trait objects used only at boundaries

## 10. Security and Isolation

### 10.1 Permission Models

#### 10.1.1 macOS TCC (Transparency, Consent, and Control)

**Required entitlements**:
- `kTCCServiceScreenCapture`: Screen recording
- `kTCCServiceAccessibility`: Input control
- `kTCCServicePostEvent`: Event posting

**User Experience**:
```
┌─────────────────────────────────────────────────────────────┐
│  "bare-cua" would like to record this computer's screen.     │
│                                                             │
│  This allows the app to capture images of the screen.       │
│                                                             │
│  [Don't Allow]                    [Open System Settings]    │
└─────────────────────────────────────────────────────────────┘
```

**Programmatic check**:
```rust
// CGPreflightScreenCaptureAccess() - macOS 10.15+
let has_access = unsafe {
    CGPreflightScreenCaptureAccess()
};
if !has_access {
    // Request access
    unsafe { CGRequestScreenCaptureAccess() };
}
```

#### 10.1.2 Windows UAC

**Manifest requirements**:
```xml
<assembly xmlns="urn:schemas-microsoft-com:asm.v1" manifestVersion="1.0">
  <trustInfo xmlns="urn:schemas-microsoft-com:asm.v2">
    <security>
      <requestedPrivileges>
        <requestedExecutionLevel level="asInvoker" uiAccess="false"/>
      </requestedPrivileges>
    </security>
  </trustInfo>
</assembly>
```

**UIPI (User Interface Privilege Isolation)**:
- Lower integrity processes cannot send input to higher integrity
- Bare-cua must run at same or higher integrity as target

#### 10.1.3 Linux Capabilities

**Required capabilities**:
- `CAP_SYS_ADMIN`: uinput device creation
- `CAP_SYS_PTRACE`: Process inspection (optional)

**Set without root**:
```bash
sudo setcap cap_sys_admin+ep /path/to/bare-cua-native
```

### 10.2 Sandboxing

#### 10.2.1 Landlock (Linux)

**LSM-based sandboxing**:
```rust
use landlock::{Ruleset, Access, PathFd};

let ruleset = Ruleset::default()
    .set_compatibility(CompatLevel::HardRequirement)
    .handle_access(Access::FS_READ_FILE)?
    .handle_access(Access::FS_WRITE_FILE)?
    .create()?;

// Restrict to /tmp
let tmp = PathFd::new("/tmp")?;
let ruleset = ruleset
    .add_rule(PathBeneath::new(tmp, Access::FS_READ_FILE | Access::FS_WRITE_FILE))?
    .restrict_self()?;
```

#### 10.2.2 Seatbelt (macOS)

**Deprecated**: Apple removed Seatbelt profiles

**Alternative**: App Sandbox (applies to apps, not CLI tools)

#### 10.2.3 Windows AppContainer

**Sandbox level**: Low integrity + capability restrictions

**Not applicable** for desktop automation (requires full access)

### 10.3 Threat Model

| Threat | Mitigation | Status |
|--------|------------|--------|
| Keylogger | User approval required (TCC) | Platform-dependent |
| Screen recording without consent | TCC prompt | macOS: enforced |
| Input injection to privileged apps | UIPI (Windows), TCC (macOS) | Partial |
| Credential exfiltration | No built-in mitigation | User responsibility |
| RCE via crafted screenshots | Image parsing validation | Implemented |
| JSON-RPC injection | Input validation | Implemented |
